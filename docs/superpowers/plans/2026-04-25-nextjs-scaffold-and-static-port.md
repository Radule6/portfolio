# Plan 1 — Next.js Scaffold + i18n Removal + Static Port

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the Vite + React 19 SPA with a Next.js 15 App Router build of the same site, English-only, no CMS yet. At the end, `npm run build && npm run start` produces a working static site that visually matches the current production site, ready for Plan 2 (Payload + Supabase).

**Architecture:** Next.js 15 App Router. All current visual components ported. Theme handled by `next-themes` (replaces the inline `<script>` hack in `index.html`). Metadata handled by Next.js native `generateMetadata` (replaces `MetaUpdater`). Projects data still hardcoded in `src/components/Projects/Projects.tsx` — Plan 2 wires it to Payload.

**Tech Stack:** Next.js 15, React 19, TypeScript 5.8, Tailwind CSS v4, next-themes 0.4+. Removed: Vite, vite-react, i18next, react-i18next.

**Spec:** `docs/superpowers/specs/2026-04-25-nextjs-migration-design.md`

**Branch:** `next-js-migration` (already created and active).

## Verification model

This is a UI/migration plan, not a logic plan. Traditional unit tests don't fit. The "test passes" gate for each task is a concrete, runnable verification:
- TypeScript: `npx tsc --noEmit` exits 0
- Build: `npm run build` exits 0 with no errors
- Runtime: `npm run dev` (or `npm run start` after build) serves the page
- Hydration: browser console clean of "hydration" / "Text content does not match" warnings
- Visual: section is visible and matches existing site

Where a task adds testable logic (e.g., `site-config.ts`), we write a small assertion. The vast majority of tasks gate on build + visual verification.

```tsx
"use client"

import { ThemeProvider as NextThemesProvider } from "next-themes"
import type { ReactNode } from "react"

export function ThemeProvider({ children }: { children: ReactNode }) {
  return (
    <NextThemesProvider
      attribute="data-theme"
      defaultTheme="system"
      enableSystem
      storageKey="radule-theme"
      disableTransitionOnChange
    >
      {children}
    </NextThemesProvider>
  )
}
```

`storageKey="radule-theme"` matches the existing localStorage key used by the inline script in the old `index.html`, so existing visitors keep their preference.

`attribute="data-theme"` matches the CSS in `app/globals.css` (`html[data-theme="light"] { ... }`).

Replace the file content with:
```tsx
"use client"

import { useEffect, useState } from "react"
import { useTheme } from "next-themes"
// keep whatever icons/SVG/styles the current component uses — only the theme-handling logic changes

export default function ThemeToggle() {
  const { resolvedTheme, setTheme } = useTheme()
  const [mounted, setMounted] = useState(false)

  // next-themes can't know the resolved theme until after hydration. Returning a static placeholder
  // avoids hydration mismatch.
  useEffect(() => setMounted(true), [])

  const isLight = resolvedTheme === "light"
  const toggle = () => setTheme(isLight ? "dark" : "light")

  if (!mounted) {
    // render a static, theme-agnostic skeleton so SSR HTML matches first client render.
    // (PRESERVE the existing visual layout — copy the JSX shell from the original ThemeToggle but
    // omit the icon swap logic.)
    return (
      <button
        type="button"
        aria-label="Toggle theme"
        className="..."  /* preserve the existing classes */
      >
        <span className="sr-only">Toggle theme</span>
      </button>
    )
  }

  return (
    /* PRESERVE the rest of the original JSX, but use `isLight` to drive icon/aria-label */
    <button
      type="button"
      aria-label={isLight ? "Switch to dark theme" : "Switch to light theme"}
      onClick={toggle}
      className="..."  /* preserve the existing classes */
    >
      {/* preserved icon JSX */}
    </button>
  )
}
```

The implementer should preserve the original component's class names, icon SVGs, and overall JSX structure — only the theme-handling logic and the `mounted` gate are new.

```tsx
import type { Metadata, Viewport } from "next"
import "./globals.css"
import { ThemeProvider } from "@/components/ThemeProvider/ThemeProvider"
import { siteConfig } from "@/lib/site-config"
import { personJsonLd } from "@/lib/structured-data"

export const metadata: Metadata = {
  metadataBase: new URL(siteConfig.url),
  title: { default: siteConfig.title, template: `%s — ${siteConfig.name}` },
  description: siteConfig.description,
  authors: [{ name: siteConfig.author }],
  alternates: { canonical: siteConfig.url },
  openGraph: {
    type: "website",
    url: siteConfig.url,
    title: siteConfig.title,
    description: siteConfig.shortDescription,
    images: [{ url: siteConfig.ogImage, alt: siteConfig.author }],
    locale: siteConfig.locale,
  },
  twitter: {
    card: "summary_large_image",
    title: siteConfig.title,
    description: siteConfig.shortDescription,
    images: [siteConfig.ogImage],
  },
  icons: { icon: "/favicon.png" },
}

export const viewport: Viewport = {
  themeColor: siteConfig.themeColor,
  width: "device-width",
  initialScale: 1,
}

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" suppressHydrationWarning>
      <head>
        <link rel="preconnect" href="https://fonts.googleapis.com" />
        <link rel="preconnect" href="https://fonts.gstatic.com" crossOrigin="" />
        <link
          href="https://fonts.googleapis.com/css2?family=Syne:wght@400;500;600;700;800&family=Outfit:wght@300;400;500;600&display=swap"
          rel="stylesheet"
        />
        <script
          type="application/ld+json"
          dangerouslySetInnerHTML={{ __html: JSON.stringify(personJsonLd) }}
        />
      </head>
      <body>
        <ThemeProvider>{children}</ThemeProvider>
      </body>
    </html>
  )
}
```

`suppressHydrationWarning` on `<html>` is required when using `next-themes`: the provider toggles `data-theme` before React hydrates, which would otherwise log a mismatch warning.
